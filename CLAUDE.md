# うちレシピ 開発ルール（全モデル・全セッション共通）

自分専用レシピ帳PWA。設計書は `../docs/`（01企画〜06バックログ）にあり、**着手前に必ず docs/06_開発バックログ.md を読むこと**。仕様判断に迷ったら 02_要件定義.md → 04_技術設計.md の順で確認。

## ユーザーについて
- プログラミング超初心者。報告は日本語で、専門用語は噛み砕いて説明する
- 大きな設計変更・お金が絡む判断は勝手にせず、ユーザーに確認する

## Fableへのエスカレーション（トークン節約運用）
このプロジェクトは普段の開発をSonnetで行い、Fableは以下の場面だけ使う。

**次のいずれかに当てはまったら、Agentツールで model: "fable" のサブエージェントに相談すること**（ユーザーの発言は要約せず**原文のまま**渡し、返ってきた判断・仕様をバックログに反映してから実装する）:

相談の作法:
- 相談文には必ず添える: ユーザー発言の原文／これまでに試したことと結果（エラー原文）／関係するファイルパス／関連する docs の該当箇所
- Fableから追加質問が返ってきたら、**新しいエージェントを立てず SendMessage で同じエージェントに返信**して対話を続ける（文脈を保つため）
- 相談で決まったこと・判明した教訓は docs/README.md の決定事項ログか本ファイルに必ず記録する（会話履歴を知識の保管場所にしない）
1. ユーザーの感想・要望が曖昧で、複数の解釈がありうるとき（例:「なんか使いにくい」「もっと便利に」）
2. 要望が既存の設計・決定事項（収益モデル、フェーズ分け、対象外機能）と矛盾する、または変更を迫るとき
3. 同じバグ修正に2回失敗したとき（状況を整理して相談）
4. ユーザーが「Fableに聞いて」と言ったとき（無条件で発動）

**サブエージェント相談では足りず、ユーザーに「Fableとのチャットを推奨」すべき場面**:
- フェーズ完了時の総合レビューと次フェーズ計画
- 収益化の開始（Pro版・広告・価格）や公開戦略など、対話しながら決める大きな判断

## 環境
- Node.js は `~/.local/node` にある。**各Bashコマンドの先頭に必ず** `export PATH="$HOME/.local/node/bin:$PATH" && ` を付ける
- スタック: React + TypeScript + Vite + Tailwind v4 + Dexie(IndexedDB) + vite-plugin-pwa + lucide-react
- devサーバー: `npm run dev` → http://localhost:5173/uchirecipe/ （ユーザーが開いていることがある。勝手に落とさない）

## コーディング規約
- UI文言はすべて `src/i18n/ja.ts` に集約（ハードコード禁止）
- 色は `src/index.css` のデザイントークン（CSS変数）のみ使用。直接色指定禁止。テーマは自動/ライト/ダーク/ブラウンの4種
- アイコンは lucide-react のみ
- スマホ縦画面基準・料理中に使う前提でボタン大きめ
- ルーティングは HashRouter。**basenameは付けない**（付けると白画面になる。過去に事故済み）
- 新フィールド追加時は任意項目にして既存データを壊さない（マイグレーション不要を保つ）

## 検証とコミット
- 完了条件: `npx tsc --noEmit` エラーゼロ ＋ `npm run build` 成功。ロジック追加時は簡易テストで動作確認
- コミットメッセージは日本語
- **push = 本番公開**（mainへのpushでGitHub Actionsが https://hapillust-cmyk.github.io/uchirecipe/ へ自動デプロイ）。動作検証が済んだ変更だけpushする

## 収益設計の前提（実装に影響する決定事項）
- 無料版=50件まで（isStarter=trueのスターターレシピは数えない）＋広告（一覧・ホーム下部のみ、料理中画面・登録画面には出さない）
- AI機能はサブスク禁止・回数券方式。サーバー必須機能（クラウド同期）はPro有料
- 個人データをサーバーに送らない（現状は完全端末内保存）
